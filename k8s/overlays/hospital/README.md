# Clinical Annotation Platform: Hospital Deployment

This directory is a self-contained deployment package for the Clinical Annotation Platform on your Kubernetes cluster.

## What you get

Four application services, all CPU-only:

- **annotation-api** (FastAPI backend, port 8001)
- **annotation-web** (Next.js frontend, port 3000)
- **pipeline-api** (NLP pipeline, port 8000) -- optional
- **pipeline-dashboard** (Streamlit, port 8501) -- optional

The platform connects to **your existing LLM serving** (e.g., vLLM serving Gemma 4) via the URL you specify in `config.env`. No LLM or GPU workloads are deployed by this package.

## Prerequisites

- A Kubernetes cluster (1.28+) with `kubectl` configured
- An Ingress controller installed (NGINX by default; configurable via `INGRESS_CLASS`)
- A StorageClass that supports `ReadWriteOnce` (about 23 GiB total, or 33 GiB with pipeline services)
- An OpenAI-compatible LLM endpoint reachable from the cluster
- A container registry you can push to (the script will build and push the images for you)
- `docker` installed locally with credentials for your registry (`docker login`)
- `envsubst` installed locally (part of the `gettext` package)
- Optional: a TLS Secret already created in your namespace

## How to deploy

1. **Edit `config.env`** with your values. The minimum to change:
   - `VLLM_ENDPOINT` and `VLLM_MODEL_NAME` (your LLM serving)
   - `DOMAIN` (where the application will be served)
   - `REGISTRY` and `IMAGE_TAG` (where to push and pull images from)
   - `STORAGE_CLASS` (or leave empty for the cluster default)
   - `TLS_SECRET_NAME` (or leave empty for HTTP only)

2. **Build and push the container images:**
   ```bash
   docker login <your-registry>   # if you haven't already
   ./build.sh
   ```
   This builds the four images from source and pushes them to `${REGISTRY}` from `config.env`. The frontend image is built with `NEXT_PUBLIC_API_URL=https://${DOMAIN}` (or `http://` if no TLS) baked in, so it knows where to reach the backend.

3. **Apply the manifests:**
   ```bash
   ./deploy.sh
   ```
   Renders the manifests into `rendered/`, shows you a preview via `kubectl kustomize`, asks for confirmation, then applies.

4. **Verify:**
   ```bash
   kubectl -n clinical-annotation get pods
   ```
   All pods should reach `Running` with `1/1 READY`.

5. **Open the UI** at `https://<your-domain>`.

## How to update

For configuration-only changes (LLM endpoint, domain, resource sizes), re-edit `config.env` and re-run `./deploy.sh`.

For application code updates, bump `IMAGE_TAG` in `config.env`, then run `./build.sh && ./deploy.sh`. Kustomize handles diffs cleanly.

## How to remove

```bash
./undeploy.sh
```

By default this keeps PersistentVolumeClaims (so session data is safe). It will prompt before deleting them.

## Troubleshooting

- **Pods stuck in `Pending`**: usually a StorageClass or quota issue. Check `kubectl describe pod <name>`.
- **annotation-api `READY 0/1`**: the readiness probe checks LLM connectivity. Verify `VLLM_ENDPOINT` is reachable from inside the cluster.
- **Ingress not working**: check `kubectl get ingress -n clinical-annotation` and verify your DNS points at the Ingress controller.

For deeper debugging:
```bash
kubectl -n clinical-annotation logs deployment/annotation-api
kubectl -n clinical-annotation describe pod -l app.kubernetes.io/name=annotation-api
```

## Architecture

```
                     +---------- Ingress ----------+
                     | /         => annotation-web |
                     | /api      => annotation-api |
                     | /pipeline => pipeline-dash  |
                     +-------------+---------------+
                                   |
        +--------------------------+--------------------------+
        |                          |                          |
  annotation-web:3000      annotation-api:8001         pipeline-dashboard:8501
                                   |                          |
                                   v                          v
                       Your LLM endpoint              pipeline-api:8000
                       (configured in config.env)
```

## Files

| File | Purpose |
|------|---------|
| `config.env` | The single configuration file you edit |
| `build.sh` | Builds and pushes the four container images using values from `config.env` |
| `deploy.sh` | Renders templates and applies manifests |
| `undeploy.sh` | Removes deployed resources (PVCs preserved by default) |
| `templates/` | YAML templates with `${VARIABLE}` placeholders |
| `rendered/` | Generated by `deploy.sh` (gitignored, safe to delete) |
